## 6. 메뉴 마스터 매칭/관리 시스템 (가장 큰 작업)

### 배경
- 마스터 "도원결의티" / POS "도원결의(HOT)", "도원결의(ICE)" — 정규화 키 다름 → 매칭 안 됨 → 죽은 메뉴로 표시
- 사용자 요청: 수동 매칭 + 마스터 청소 도구

### 정규화 강화 — `stripVariantSuffix`
매칭 키 생성용으로 분리 강화:
- 공백·괄호·구두점 모두 제거 + 소문자
- "말차 티라미수 미니" / "말차 티라미수(미니)" → 둘 다 `말차티라미수미니` 자동 매칭
- `normalizeProductName`은 표시용(공백 유지)으로 분리

### product_aliases 테이블
```sql
CREATE TABLE IF NOT EXISTS product_aliases (
  id bigserial PRIMARY KEY,
  product_id text NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  pos_name text NOT NULL UNIQUE,
  created_at timestamptz DEFAULT now()
);
ALTER TABLE product_aliases DISABLE ROW LEVEL SECURITY;
CREATE INDEX IF NOT EXISTS idx_product_aliases_pos_name ON product_aliases(pos_name);
```

### MatchMenuButton 모달
- 죽은 메뉴 / 마스터 관리 행 옆 ✏️
- 전체 POS 이름 + 매칭 상태 표시:
  - 초록 chip = 자기 마스터에 이미 매칭 (수동/자동)
  - 회색 chip = 다른 마스터에 매칭됨
- **multi-match**: 클릭 후 모달 안 닫힘. 여러 POS 한 번에 매칭. 닫기 시 새로고침
- 자기 마스터 매칭 다시 클릭 = 해제 (toggle)
- 다른 마스터에 매칭된 거 옮기려면 confirm

### MasterManager — 메뉴 마스터 관리 표
`/upload 메뉴 마스터` 탭에 노출:
- 1열: 마스터명 (정렬 토글, asc/desc)
- 2열: 매칭된 POS 이름 chips (초록=수동 / 회색=자동) + ✏️
- 3열: 활성/비활성 토글
- 검색 + "비활성 포함" + **"정리 필요만 보기"** 필터
- 행 라벨: ⚠ 키 충돌 / ◌ 매칭 없음

### 죽은 메뉴 옆 🗑 비활성화
- `DeactivateMenuButton` — confirm 후 PATCH /api/products
- 마스터 관리 페이지 안 가도 단종/중복 청소 가능

### alias 우선 매칭 (`getDeadMenus`)
- raw `pos_name` → master_key 사전을 자동 정규화보다 먼저 시도
- 점장이 명시한 매핑이 항상 우선

### `/menu` 디폴트 변경
- range: `4w` → **`all`** (전체 12개월)
- 죽은 메뉴 정렬: 판매량 낮은 순 → **마지막 판매 오래된 순**, null(한 번도 안 팔림) 최우선

## 7. 점장과의 마스터 청소 케이스별 협의

실제 데이터 보면서 케이스별 처리 정립:

| 케이스 | 처리 |
|---|---|
| 정규화 키 충돌 (도원결의티 + 도원결의(HOT)) | 한 마스터에 alias 묶고 다른 비활성화 |
| 같은 stripped 키 마스터 다수 (교토 말차세트 / (HOT) / (ICE)) | "통일 이름" 마스터만 활성, 나머지 비활성화 (alias 등록 불필요 — 자동 매칭으로 흡수) |
| 정규화로 매칭 안 됨 (더치커피(ICE Only) / 더치커피(ICE)) | "통일 이름" 마스터에 두 POS alias 등록 + 다른 비활성화 |
| POS에 없는 마스터 (더치라떼(ICE Only)) | 단종 처리 — 비활성화 |

### 규칙 한 줄
- **chip "—" 인 활성 마스터 = 정리 후보**
- 자동 매칭이 잘 됐으면 alias 등록 불필요
- 비활성화는 데이터 손실 X (마킹만)
- 4/23 합의 적용: HOT/ICE 통일 (티 제외)

## 13. 데이터 흐름 정립 (점장 이해 도움)

대화 중 점장이 "마스터 손대도 되는 거?" 헷갈림 → 그림 그려 설명:

### 두 데이터 소스의 역할
- `daily_sales` (POS) = 실제 팔린 기록
- `products` (마스터) = 메뉴 카탈로그 (분석 대상 정의)

### 매칭 단계
1. **자동 매칭** — `stripVariantSuffix`로 정규화 후 키 비교
2. **수동 매칭 (alias)** — 정규화로 안 잡히는 케이스
3. **마스터 청소** — 중복/단종 비활성화

### 비활성화 ≠ 삭제
- `is_active = false` 마킹만
- POS 거래 데이터 안 건드림 (다른 활성 마스터에 자동 매칭)
- 다시 활성화 가능

## 15. 느낀 점

- **콕핏의 마지막 퍼즐 = 첫 줄 요약 통일**: 운영 페이지(대시보드/주간/지출/이익/메뉴) 모두 첫 줄에 InsightBanner 형태로 통일. 점장이 어느 페이지를 열어도 "지금 알아야 할 것"을 한눈에. 디테일은 그 다음. 매일 쓰는 도구의 핵심 패턴.

- **"가장 큰" 인사이트의 함정**: `/expenses`에서 "가장 큰 지출 = 인건비"는 매번 같은 답. 가치 약함. **변화량 기반("전월比 가장 많이 증가한 카테고리")** 으로 바꾸니 즉시 액션 가능한 정보로 변환. **절대값 < 변화량**.

- **마스터 청소는 데이터 도메인 판단**: 시스템이 "도원결의티 = 도원결의(HOT)인가?" 자동 결정 못 함. 점장이 결정. 시스템은 그 결정을 쉽게 표현/실행할 수 있는 도구만 제공. UI = 권한 위임 + 가이드.

- **"같은 메뉴" 케이스가 의외로 많음**: POS export에 (HOT)/(ICE)/(미니) 변형이 별도 마스터로 들어옴. 점장이 모르고 사용. 분기 1회 청소가 자연스러운 운영 루틴.

- **만들지 않는 결정도 결정**: 사장 보고 모드 처음엔 "필요" → 다시 보니 `/profit`이 본질을 가짐. 별도 모드 폐기. 인쇄도 만들었다 제거. **만들기보다 안 만드는 결정이 더 어렵고 더 가치 있을 때가 있음**.

- **모바일 대응의 적정선**: 콕핏은 폰에서 fully 작동, 관리 도구는 PC 전용. **작업 성격에 따라 분리**가 자연스러움. 모든 화면을 모바일 친화 만들 필요 X. 점장이 폰으로 매일 보는 것 / PC에서 분기 청소하는 것 — 다른 도구.

- **점장 이해 = 점장 자신감**: "마스터 손대도 되나?" 헷갈림을 데이터 흐름 그림으로 설명. 비활성화 ≠ 삭제 명확화. 이해한 후 청소 작업 자신감 ↑, 더 적극적. **시스템 mental model을 사용자와 공유하는 것이 UI 디자인만큼 중요**.

- **자기비판이 거꾸로 갈 수 있다 (4/25 노트 인용)**: 사장 보고 모드 — "후퇴" 하다가 다시 "본질이 이미 있으니 안 만든다" 결정. 매번 *제약과 가정을 다시 체크*. 보수적 후퇴 = 안전 아님.

- **점장과의 페어 작업 가치**: 데이터 케이스(도원결의티, 교토 말차세트, 더치커피, 핫초코)를 함께 보면서 처리 정립. 추측 X, 실제 데이터로 확인. 시스템 사용 워크플로가 점장 도메인 판단과 맞물려 정착됨.
