## 11. 느낀 점

- **자기비판이 거꾸로 갈 수 있다는 발견**: "보수적으로 후퇴 = 안전"이 아님. 사용자가 제약을 풀어주니 원래 강한 안이 더 정확했다. 
  자기비판은 *동의받은 제약 안에서* 의미 있고, 제약 자체를 의심해야 할 때 보수적 후퇴는 잘못된 방향.

- **데이터 모델이 "같은 행동"의 결과를 자동으로 모아주지 못하면 매칭은 깨진다**: counterpart 누락 → memo 기반 매칭 실패. 4/23 노트에서 이미 "counterpart 컬럼 고려"라고 적어놓고 안 했던 게 결국 발목 잡음.
  → 미래의 자기 자신에게 보내는 신호: "고려"라고 적은 건 그게 맞다는 뜻이니 그냥 해라.

- **점장 사용 빈도 ≠ 도메인 직교 그리드**: 4/20에 "각 페이지가 다른 질문에 답한다"로 도메인 분할, 4/23에 "시간축 vs 원인"으로 IA, 4/25에 "운영/결산/관리"로 빈도 기반 IA. 세 번째가 가장 사용 흐름과 일치함. 도메인이나 시간축으로 가르는 건 시각적으로 깔끔하지만 점장 머리는 "매일 / 매주 / 월말 / 가끔" 으로 작동.

- **신호등 임계값은 점장에게 묻지 말고 데이터에서 찾는다**: 절대 임계 35%/45% 같은 업종 평균은 카페마다 안 맞음. 전월比 변화량 기준은 운영 데이터가 자체적으로 정의하는 정상 범위라 더 안전. 절대값은 데이터 충분히 쌓인 후 도입.

- **메뉴 마스터 = 죽은 메뉴 분석의 정답 키**: POS 거래 데이터에서 유니버스를 "유추"하던 게 본질적 약점이었음. 마스터(products)가 진실의 원천(SoT)이 되니 단종 메뉴 노출 / 신메뉴 0건 표시 / 카테고리 정확 매핑 모두 자연스럽게 풀림. 4/23 노트의 "counterpart 컬럼" 교훈과 같은 패턴 — *유추하지 말고 사실을 저장하라*.

- **IA 한 번 더 흔들기**: 점장 적응 비용 걱정해서 후퇴했다가 사용자가 "걱정 X"라고 풀어준 게 결정적. 좋은 IA 한 번에 만들 수 없고 사용 데이터 보면서 흔드는 게 맞음. 4번째 IA 재편이 와도 받아들일 자세.
