## 4. Step B — 분류 체계 확장

### 새 카테고리 2개 추가
- **`equipment`** (설비투자) — 일회성 자본 지출 (테이블/스피커/반죽기 등)
- **`excluded`** (제외) — 집계에서 완전 제외 (박기선 개인 소비 등)

### 파싱 규칙 대폭 확장 (`bank/route.ts`)

**재료비 (ingredients)**
- 기존: 원두/말차/우유/햄/원두값/말차값
- 추가: 재료/비품/포장/페트병/드립백
- 수취인 매칭: **홍인호** (`INGREDIENT_COUNTERPARTS`)

**고정비 (fixed)**
- 기존: 전기세/지방세/세금/임대료/보험료/근로소득세
- 추가: 전기**요금**/직장합산/건강보험/국민연금/회계사/지코드

**설비투자 (equipment) — 신규**
- 키워드: 테이블/스피커/반죽기/액자/소분기/더치기구

**제외 (excluded) — 신규**
- 수취인 매칭: **박기선** (`EXCLUDED_COUNTERPARTS`)

### 분류 순서 조정
`excluded → equipment → ingredients → fixed → card`
- `equipment`를 `ingredients`보다 먼저 체크 ("드립백소분기"처럼 키워드 겹치는 케이스 대응)

### UI/쿼리 업데이트
- `PreviewTable`, `DataEditor` — 드롭다운에 `equipment`/`excluded` 추가
- `DeficitBreakdown` — equipment 필드 추가 (sky 색상)
- `ExpenseTrendChart` — equipment 라인 추가
- `ExpenseItemList`, `ExpenseDonut` — equipment 섹션/라벨 추가
- `queries/dashboard.ts`, `queries/expenses.ts`, `queries/weekly.ts` — **`excluded` 집계 제외** 필터
- `save/route.ts` — `total_expense` 계산 시 excluded 제외

## 8. 오후 세션 — POS 상품 단위 전환, 메뉴 분석 신설, /이익 전략 뷰 등

> 같은 날 오후에 POS 실제 데이터 도착 후 크게 확장. Step B~D 상당 부분 해결됨.

### POS 파서 — 상품 라인 단위로 전환
- 이전: 카테고리 단위 집계만 (`daily_sales.date, category, amount`)
- 신규: 상품 라인 단위 저장 (`date, order_id, line_no, product_name, category, quantity, amount, order_time`)
- **스키마 변경**
  ```sql
  ALTER TABLE daily_sales
    ADD COLUMN IF NOT EXISTS product_name TEXT,
    ADD COLUMN IF NOT EXISTS quantity INTEGER DEFAULT 0,
    ADD COLUMN IF NOT EXISTS order_id TEXT,
    ADD COLUMN IF NOT EXISTS line_no INTEGER,
    ADD COLUMN IF NOT EXISTS order_time TIMESTAMPTZ;
  ALTER TABLE daily_sales DROP CONSTRAINT IF EXISTS daily_sales_date_category_key;
  ALTER TABLE daily_sales ADD CONSTRAINT daily_sales_line_key
    UNIQUE (date, order_id, line_no, source);
  ```
- **파일명 제약 제거** — POS 파일 내부 `주문 시간` 컬럼에서 날짜 자동 추출
- **취소/반품 처리 규정** (사용자 확인): 취소는 완전 제외, 반품은 엑셀에 이미 음수로 저장되어 있어 그대로 합산 → 총매출 50,417,900 일치

### 카테고리 체계 재정립 (메뉴판 기준 9개 + etc)
`coffee` · `drip_coffee` · `dutch_coffee` · `beverage` · `dessert` · `season` · `matcha` · `ade` · `tea` · `etc`
- 우선순위: etc → dessert → season → matcha → ade → tea → dutch_coffee → drip_coffee → coffee → beverage(fallback)
- "바스크 치즈케익(말차)" 같은 케이스는 dessert로 귀결 (dessert가 matcha보다 먼저)
- `이벤트용 봄딸기주스` → etc에서 제외하고 season으로 이동 (사용자 요청)

### Supabase PostgREST 1000-row 제한 회피
- **증상**: 상품 라인 단위 저장 후 월 ~3500행 중 1000행만 읽혀 `monthly_summary.income` 심각하게 저평가 (3,189만 표시 / 실제 5,041만)
- **해결**: `lib/supabase/fetchAll.ts` 헬퍼로 `.range()` 페이지네이션
  ```ts
  const all = await fetchAllRows<T>((from, to) => query.range(from, to))
  ```
- 적용: `save/route.ts` POS 재계산, `queries/weekly.ts`, `queries/income.ts`, `queries/dashboard.ts`

### Bank 파서 — tx_time + balance_after upsert

## 9. 점장/오너 관점 리뷰 & 다음 우선순위

### 현재 시스템 평가
- **강점**: 데이터 신뢰도, 질문 기반 네비게이션, 주간보고 전용 뷰, 액션 인사이트(요일×시간 히트맵·BEST10 순위변동 등), 브랜드 톤 일관성
- **약점**:
  1. "그래서 뭘 해야 해?"까지 연결이 약함 (액션 제안 문장 없음)
  2. **원가 데이터 부재** → 상품별 마진 불가 → 메뉴 개선 의사결정 근거 약함
  3. 비정상 매출일 감지 없음 (anomaly detection)
  4. 대여금·excluded 연동 수동 (놓치면 수치 왜곡)
  5. 업로드 상태/갱신 리마인더 없음
  6. 현금/통장 잔액 추적 없음 (balance_after 미활용)
  7. 오너/점장 권한 분리 없음
