## 2. 핵심 개념 분리 — 원가 ≠ 마진

대화 초반 사용자 질문: "원가율 계산을 한 번 해보려고 하거든? 일단 아메리카노만 생각해볼까."

이전 대화에서 "가격·구매날짜·매출 엮어서" 가다가 점장 컨펌 위해 멈췄던 컨텍스트. 오늘 단순화 결정:

- **원가**: 메뉴 한 잔에 들어가는 재료비 (재료 단가 × 사용량의 합). 시점·매출 무관
- **마진/원가율**: 원가를 판매가/매출과 엮을 때 비로소 필요

→ 이번 작업의 **범위는 "메뉴별 원가까지"**. 마진은 다음 단계.

## 3. 결정사항 (점장 합의 전제)

각 결정을 사용자와 **하나씩** 합의해서 plan에 녹임. 결정의 *왜*를 잊지 않기 위해 기록:

### 3.1 HOT/ICE 통합
- 처음엔 분리 고려 (PDF의 카페라떼 ICE 우유량이 미명시 → 다를 수 있음)
- 사용자: "PDF가 중복 생략한 거야. 그람수는 같고 핫은 스팀, 아이스는 차가운 그대로"
- → `stripVariantSuffix()` 정규화 결과를 레시피 키로 사용 (HOT/ICE 한 레시피)
- 물·얼음 원가는 0 (카페에서 물값 안 나가고 얼음 무시)

### 3.2 사이즈 변형 없음
- 1메뉴 = 1레시피

### 3.3 매장 vs 포장 = 레시피 + 포장세트 합산 모델
- 메뉴 자체 레시피는 매장 기준
- 포장 시 추가되는 컵·뚜껑·홀더 등은 별도 "포장 세트"로 합산
- 단순화: 메뉴별로 어떤 포장 쓰는지 일일이 매핑 안 함 → **카테고리 × 온도 매핑** (음료-HOT는 핫컵세트, 음료-ICE는 콜드컵세트 등)
- 단, **말차류 매장/포장 사용량이 PDF에선 다름**: 매장 H 파우더10g vs 포장 H 25g. 점장 컨펌 필요 — 일단 매장 HOT 기준 입력, 차이 처리는 점장 미팅에서

### 3.4 재료 두 종류 — 매입 / 수제
**핵심 깨달음**: PDF에 "흑임자크림(흑임자20+생60)", 말차티라미수 베이스 등 *수제 중간재료*가 있음. 디저트·청은 전부 수제(점장이 한 번 만들 때 N개 분량 만들고 메뉴 1잔에 X g씩 사용).

→ ingredients 테이블에 `kind` 컬럼:
- `purchased`(매입): 단가는 `ingredient_prices`에 직접 입력
- `made`(수제): 단가는 `sub_recipes`에서 자동 계산 = 구성재료비 합 ÷ 1배치 산출량

**대안 검토 → 거부**:
- 옵션 B: 점장이 직접 1g 환산 단가 입력 → 사용자: "너무 까다로워. 너무 귀찮은 작업이라 안 될 것 같음"
- 즉 자동 계산이 필수

### 3.5 서브레시피 재귀 깊이 1단계
- 수제재료의 구성요소는 매입재료만 허용 (다른 수제재료 못 가짐)
- 사용자: "베이스 안에 크림이 들어간다는 게 무슨 말인지 이해를 못했어" → 그런 케이스 없음 = 1단계로 충분

### 3.6 단가 이력은 스키마 유지, 입력은 선택
- `ingredient_prices`에 `effective_date` 컬럼
- 현실: 원두/말차/햄/우유는 현금 결제라 통장에 날짜+금액 보여 이력 의미 있음. 카드결제 포장재는 통합 결제로 시점 추적 불가
- 즉 행이 1개든 여러 개든 조회 로직 동일 (`effective_date <= asOfDate, 최신`)
- **모든 첫 입력의 적용시작일은 `2025-12-15` 통일** (시스템 도입 기준일)

### 3.7 누락은 "원가 미등록" 뱃지
- 레시피 미등록 메뉴 / 단가 미등록 재료 포함 메뉴 → 0원 처리 X (평균 왜곡)
- 명시적으로 "원가 미등록" 표시, 평균 집계에서 제외
- 반환 타입: `{ status: 'ok' | 'no_recipe' | 'missing_price', baseCost?, packagingCost?, totalCost?, missingIngredients? }`

### 3.8 포장세트 매핑 = 카테고리 × 온도
- DB에서 확인된 카테고리 영문 키 (실제 daily_sales.category):
  ```
  coffee(26), tea(26), dessert(17), matcha(13), beverage(10),
  ade(4), etc(4), drip_coffee(2), dutch_coffee(2), season(2)
  ```
- HOT/ICE 컵이 다르므로 한 카테고리에 두 세트 (디저트는 온도 무관 = null)
- UNIQUE(product_category, serve_temp)

### 3.9 UI 위치 = `/recipes` 새 페이지, 사이드바 "원가"
- "관리" 그룹에 `/staff` 옆 추가 (사이드바 라벨은 `원가` 한 단어 — 다른 항목 톤과 일치)
- 권한 시스템 없음 — OwnerToggle은 권한이 아니라 "대표 개인 거래 집계 포함 토글" (확인 완료)

## 5. 데이터 모델 (확정)

### 5.1 ingredients (재료 마스터)
```
id            uuid PK
name          text UNIQUE
unit          text                   -- 'g' | 'ml' | '개' | '펌프' | '회' | '조각'
kind          text                   -- 'purchased' | 'made'
is_active     boolean DEFAULT true
created_at    timestamptz
```

### 5.2 ingredient_prices (단가 이력 — 매입 재료만)
```
id              uuid PK
ingredient_id   uuid → ingredients(id) ON DELETE CASCADE
unit_price      numeric                -- 원/단위
effective_date  date
created_at      timestamptz
INDEX (ingredient_id, effective_date DESC)
```
조회: `WHERE ingredient_id=? AND effective_date <= asOfDate ORDER BY effective_date DESC LIMIT 1`

### 5.3 recipes (메뉴 레시피)
```
id                       uuid PK
product_name_normalized  text                   -- stripVariantSuffix() 결과
ingredient_id            uuid → ingredients(id)
quantity                 numeric
created_at               timestamptz
UNIQUE (product_name_normalized, ingredient_id)
INDEX (product_name_normalized)
```
- products.id 아닌 정규화 이름을 키로 → HOT/ICE 자동 통합
- 매칭 시 `lib/menu-utils.ts:stripVariantSuffix` 재사용

### 5.4 sub_recipes + sub_recipe_items (수제 재료 배합)
```
sub_recipes:
  id                    uuid PK
  output_ingredient_id  uuid → ingredients(id) UNIQUE   -- kind='made' 만
  output_quantity       numeric                         -- 1배치 산출량
  notes                 text
  created_at            timestamptz

sub_recipe_items:
  id              uuid PK
  sub_recipe_id   uuid → sub_recipes(id) ON DELETE CASCADE
  ingredient_id   uuid → ingredients(id)               -- kind='purchased' 만 (1단계 제한)
  quantity        numeric
  UNIQUE (sub_recipe_id, ingredient_id)
```

### 5.5 packaging_sets + packaging_set_items
```
packaging_sets:
  id                uuid PK
  name              text                          -- '커피 HOT 포장', '디저트 포장'…
  product_category  text                          -- daily_sales.category 키
  serve_temp        text                          -- 'hot' | 'ice' | null
  is_active         boolean DEFAULT true
  created_at        timestamptz
  UNIQUE (product_category, serve_temp)

packaging_set_items: (sub_recipe_items와 동일 구조)
```

## 6. 원가 계산 로직

`lib/supabase/queries/recipe-costs.ts` (신규):

```ts
async function getMenuCost(
  productNameNormalized: string,
  opts?: { asOfDate?: string },
): Promise<{
  status: 'ok' | 'no_recipe' | 'missing_price'
  baseCost?: number
  packagingCost?: number
  totalCost?: number
  missingIngredients?: string[]
}>
```

- `baseCost` = Σ(레시피.quantity × 재료 단가)
  - 매입 재료 단가 = `ingredient_prices`에서 그 시점 최신값
  - 수제 재료 단가 = (Σ sub_recipe_items.quantity × 매입재료 단가) / output_quantity
- `packagingCost` = (메뉴 카테고리, 메뉴 온도) 매핑된 포장세트 합. 매핑 없으면 0
- 메뉴 온도 추출: products.name에서 HOT/ICE suffix (구체 정규식은 코드 작업 시 stripVariantSuffix 보고 결정)

`getAllActiveMenuCosts(asOfDate?)` — products.is_active=true 일괄 계산.

## 7. 엑셀 입력 양식 (점장용)

### v1 → v2 변경 이유
v1은 점장이 단가를 1원/단위 직접 입력하는 구조. 사용자 피드백:

> "한 번 뭘 구매할 때 용량이 몇 인지, 그걸 몇 세트 샀는지, 가격은 얼마인지 이렇게가 재료 시트에 들어가야 하지 않을까. 점장이 다 단위 당 원가를 적긴 힘들잖아."

→ v2에서 **구매현황 입력 → 단가 자동계산**으로 전환.

### v2 시트 구성 (`notes/레시피_템플릿_v2.xlsx`)

**시트1 재료** (총 60행: 매입 47 + 수제 13)
| 재료명 | 사용단위 | 종류 | 1세트 용량 | 세트 수 | 총 가격(원) | 단가(자동) | 적용시작일 | 비고 |

- 단가 = `=IFERROR(F{row}/(D{row}*E{row}),"")` 엑셀 수식
- 적용시작일 모두 `2025-12-15` 사전입력
- 예) "원두 1kg봉지 2개 70,000원" → 1세트용량 1000, 세트수 2, 총가격 70000 → 단가 35원/g 자동
- 예) "바닐라시럽 1병 = 200펌프, 30,000원" → 1세트용량 200, 세트수 1, 총가격 30000 → 150원/펌프
- 수제재료는 단가 행을 "(서브레시피 자동계산)"으로

**시트2 서브레시피** (29행)
| 수제재료명 | 1배치 산출량 | 구성재료명 | 사용량 | 비고 |
- 흑임자크림만 PDF에서 정확 추출 (1배치 80g, 흑임자 20g + 생크림 60g)
- 나머지는 구성·산출량 모두 ❓ — 점장 입력 필수

**시트3 레시피** (DB 활성메뉴 66개 + 레시피 행 다수)
| 메뉴명 | 카테고리 | 재료명 | 사용량 | 비고 |
- DB 정확한 이름 기준 (POS export와 동일)

## 11. 느낀 점

- **"메뉴 하나 = 시트 하나" 오해를 빨리 해소해야 한다**: 사용자가 처음 엑셀 양식 본 후 "메뉴 하나 = 시트 하나?"라고 물음. Long format(한 시트 모든 메뉴, 한 행 = 한 (메뉴, 재료) 튜플)이 직관적이지 않을 수 있음 → 첫 양식 보낼 때 PDF 데이터 일부로 *살아있는 예시* 보여주면서 설명하면 효과적이었음.

- **사용자가 데이터 현실을 가장 잘 안다**: 단가 이력 관리에서 "원두/말차/햄/우유는 현금 결제라 통장에 날짜+금액 보임. 카드결제 포장재는 통합결제로 추적 불가" — 이런 운영 실태는 코드만 봐선 모름. 모델 결정 전에 *데이터가 어떻게 들어오는지* 사용자에게 물어보는 게 결정적.

- **점장 입력 부담 = 모델 정확도의 천장**: 옵션 B(점장이 1g 환산 단가 직접 입력) 거부 → 자동 계산 강제. 이는 모델 복잡도를 약간 올리지만, 점장이 안 채우면 0이 되니 정확도가 망가짐. 항상 "사용자 입력 부담이 정보 정확도를 낮추지 않는가" 점검.

- **DB 우선 vs PDF 우선**: 처음엔 PDF 메뉴명으로 엑셀 만들었다가 "DB 메뉴명과 다르면 매칭 실패" 발견 → DB 메뉴명 기준으로 재생성. 시스템 통합 작업에선 *진실의 원천(DB)* 우선이 옳음. PDF는 정보 소스, DB는 매칭 키.

- **계획 단계의 가치**: 오늘 코드 0줄. 그러나 plan 파일이 매우 두꺼워졌고 점장 미팅에서 물을 항목이 명확. 코드 시작하면 `claude code "plan 파일 따라 구현해줘"` 정도로 진행 가능. 4/25의 IA 대수술이 그랬듯, **합의 + 문서화 → 구현은 거의 자동**.
