## 3. DB 마이그레이션 — 5테이블 + 부속 2개

```sql
-- ingredients (재료 마스터) + ingredient_prices (이력)
-- recipes (메뉴 레시피, product_name_normalized 키)
-- sub_recipes + sub_recipe_items (수제재료 배합, 1단계)
-- packaging_sets + packaging_set_items (카테고리×온도)
```

설계 결정:
- **bigserial PK** (프로젝트 컨벤션). 4/28 plan은 uuid 였지만 다른 테이블과 일치시킴
- `payment_method` 컬럼 — 엑셀 결제방식 → DB 매핑
- `(ingredient_id, effective_date)` UNIQUE — 같은 날짜 단가 중복 방지 (재업로드 갱신)
- `packaging_sets`는 `serve_temp = NULL` 허용용 partial unique index (`COALESCE(serve_temp, '')`)
- `upload_history.file_type` CHECK 제거 (recipe 타입 추가 위함)

RLS 비활성화 (프로젝트 원칙).

## 4. 업로드 파이프라인 — `app/api/upload/recipe/route.ts`

### 단순화
4시트라 미리보기 단계 없이 **파싱 + 저장 일체형**. 결과 stats만 반환.

### 파서 핵심
- 단가는 (1세트용량 × 세트수)에서 **직접 재계산** — 엑셀 수식 캐시 무시 (생성 시 캐시 안 됨)
- ❓ / 빈 값 / `📌`·`예시` 메모 → 스킵 (안전)
- 메뉴명은 `stripVariantSuffix()`로 정규화 (HOT/ICE 통합 키)
- "↓ 점장 자유 추가" 마커 행 스킵

### 저장 순서
1. `ingredients` upsert (name 충돌)
2. `ingredient_prices` upsert (`ingredient_id, effective_date` 충돌)
3. `sub_recipes` upsert (`output_ingredient_id` 충돌) + items 전체 삭제 후 재삽입
4. `recipes` 메뉴 단위 삭제 후 재삽입 (구성 변경 반영)
5. `packaging_sets` 카테고리×온도로 조회/생성 + items 삭제 후 재삽입

### 부분 입력 처리
- 단가 누락이면 그 재료의 `ingredient_prices` 행 안 들어감
- 잔당 사용량 ❓이면 그 메뉴의 그 라인만 스킵
- 다음 업로드 시 같은 양식 그대로 → 갱신

## 5. 원가 계산 — `lib/supabase/queries/recipe-costs.ts`

### `MenuCost` 타입
```ts
type CostStatus = 'ok' | 'no_recipe' | 'missing_price'

interface MenuCost {
  productId, productName, productCategory, servingTemp, price
  status, baseCost?, packagingCost?, totalCost?, margin?, costRatio?
  missingIngredients?: string[]
}
```

### 계산 흐름
```
products(active) → 각 메뉴
  ├─ recipes 조회 by stripVariantSuffix(name)
  │   ├─ 없으면 no_recipe
  │   └─ 있으면 각 ingredient_id로 단가 조회
  │       - kind='purchased': ingredient_prices 시점별 최신
  │       - kind='made': sub_recipes 재귀 (1단계) — 각 구성재료 × 단가 / output_quantity
  ├─ packaging_sets by (category, temp) — 매칭 없으면 0원 (실패 X)
  └─ 누락 1개라도 있으면 missing_price + missingIngredients 리스트
```

### 카테고리 매핑
products 테이블에 카테고리 컬럼 없음 → `daily_sales`에서 distinct (`stripVariantSuffix(name)` → category)로 빌드.

### `summarizeCosts(costs)`
- 평균 원가율 = `ok` 메뉴만으로 (미등록은 0원 처리 X — 평균 왜곡 방지)

## 10. 1봉 vs 10봉 단가 동치성 (대화 메모)

점장이 "한 번 살 때 보통 10봉" 이라 했고 4월 통장엔 80만 찍힘.
- 1봉 35,000원 / 1세트용량 1000 / 세트수 1 = 35원/g
- 10봉 350,000원 / 1세트용량 1000 / 세트수 10 = 35원/g
- **단가 계산은 완전 동일**, 미래 "봉지당 회수율" 화면도 영향 없음 (1봉 = 1세트용량 × 단가)
- "한 번 매입 시 N봉" 정보는 세트수에 들어가는데 정확히 모르면 1로 두는 게 깔끔. 점장 컨펌 후 갱신.

## 12. 느낀 점

- **부분 입력 설계의 가치**: 점장 데이터 0% 상태에서도 시스템이 "0원 처리 X / missing_price 명시" 로 깨끗이 동작. 평균 집계가 ok 메뉴만으로 계산되니 부분적이라도 신뢰할 수 있는 숫자가 나옴. 이 설계가 4/28 plan §3.7에서 이미 정해진 게 결정적.

- **계획 → 구현이 거의 자동**: 4/28의 plan 파일 + v5 엑셀 양식이 진실의 원천이 되어 코드 작업이 매끄러웠음. 5테이블 스키마/파서 흐름/원가 계산식이 모두 plan에 적혀있어 "구현 = 옮기기"에 가까움. 4/25 IA 대수술과 같은 패턴 — *합의·문서화 → 구현은 거의 자동*.

- **점장 PC의 Claude Code = 미지원 부분의 백업**: 시스템이 입력 양식만 받고 직접 도와주진 못함. 그 빈 공간을 점장 PC의 Claude Code가 메움 — 가이드 파일 self-contained로 만들고 CLAUDE.md에 포인터 두니, 점장이 막힐 때마다 즉답 가능. *문서가 곧 백업 시스템*.

- **메시지 분류의 위력**: `no_recipe / missing_price / ok` 세 상태 + missingIngredients 리스트만으로 점장이 "왜 이거 안 떴어? → 어디 가서 채우면 돼" 까지 자기 해결 가능. 화면이 자체적으로 다음 행동을 가리키는 구조.

- **샘플 가짜 숫자 = 양날의 검**: v5에서 자몽청·말차크림 샘플로 패턴 학습 도와주려고 추가했는데, *실데이터로 오인될 위험* 있음. 메모 마커(`📌`)로 파서 자동 스킵 + 테스트 파일에선 ❓로 비움 — 이중 안전. 하지만 점장이 실제 입력하면서 메모 지우고 숫자만 두는 케이스는 여전히 위험. 추후 입력 끝난 v5 받으면 검증 필요.

- **단가 계산의 단위 불변성**: "1봉 vs 10봉" 케이스에서 결과 동일. 단가 = 총가격/(용량×세트수)이라 단위 단위가 무엇이든 결과는 같음. 이게 양식 설계의 견고함.

- **테스트 파일을 만들고 직접 검증한 가치**: 원두 1봉만 채운 테스트 파일로 검증해보니 의도대로 동작. 빈 시스템 → 부분 데이터 → 결과 검증의 사이클이 빨라야 다음 단계 자신감 ↑.
