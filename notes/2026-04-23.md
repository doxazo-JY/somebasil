## 1. 점장과의 논의에서 확정된 것들

### 카테고리
- **재료비를 현금/카드로 분리**
  - 현금(정기): 홍인호(원두)·김인성(말차)·한성욱(우유)·소금집
  - 카드(비정기): 쿠팡·이마트·씨유·원두값·포장·페트병 등
  - 점장 관점: 매달 고정적으로 나가는 재료비 vs 그때그때 쓰는 재료비를 구분해서 보고 싶다
- **전기 키워드 단순화**: `전기세`/`전기요금` → 그냥 `전기` 한 단어
- **고정비 키워드 우선체크**: 지방세·등록면허세가 `excluded`보다 먼저 체크되게 순서 조정
  - 이유: `지방세_인천계양구(박기선)` 같은 수취인이 excluded(박기선) 규칙에 먼저 매칭되던 버그

## 2. /설정 페이지 대수술

### 기존 개념 버림
- "대여금 CRUD" 탭 통째로 제거 (`loans` 테이블 DROP, `LoanSection` 삭제)
- 이유: 대여금 장부는 별도로 유지할 가치가 낮음. 통장에 이미 찍힌 상환 거래를 **지출에서 빼는 수단**이 본질.

### 신규 구조
```
설정
├─ 대표 개인 거래 on/off 토글
│   └─ excluded 카테고리 전체를 집계에 포함/제외
└─ 수동 수입/지출 조정
    └─ 자동 분류 불가능한 것을 월별 가감
```

### 왜 수동 조정이냐
- 장천섭 1,000만 빌리고 500만 상환 케이스
- 근데 실제 통장엔 **400만만 출금** (100만은 카페 메뉴로 제공)
- 100만원치 음료 = POS엔 매출로 이미 찍힘
- 이런 엣지케이스를 자동 매칭으로 풀려면 복잡도 폭발
- 그냥 **"수동 조정" 범용 기능**으로 흡수 → 월별 집계 `monthly_summary`에 자동 반영

### 테이블 설계
```sql
-- 범용 키/값 설정 (대표 토글 외에도 확장 가능)
CREATE TABLE system_settings (
  key text PRIMARY KEY,
  value jsonb NOT NULL,
  updated_at timestamptz DEFAULT now()
);

-- 수동 조정
CREATE TABLE manual_adjustments (
  id bigserial PRIMARY KEY,
  date date NOT NULL,
  type text CHECK (type IN ('income', 'expense')),
  direction text CHECK (direction IN ('add', 'subtract')),
  amount integer CHECK (amount > 0),
  memo text,
  created_at timestamptz DEFAULT now()
);
```

### 집계 통합 — `lib/supabase/recalc.ts`
공용 `recalcMonthlySummary(year, month)` 유틸 신설:
- 수입 = daily_sales(POS 우선) + 수동 수입 조정
- 지출 = monthly_expenses(excluded 제외, 토글 on이면 포함) + 수동 지출 조정

호출 지점 통일:
- `app/api/settings/route.ts` — 토글 변경 시 `recalcAllMonths`
- `app/api/adjustments/route.ts` — 조정 추가/삭제 시 해당 월만
- `app/api/upload/save/route.ts` — 업로드 완료 후 영향받은 월
- `app/api/upload/edit/route.ts` — 재분류/삭제 시 해당 월

중복 로직(inline recalc 여러 곳에 있던 것) 전부 제거됨.

## 3. 사이드바 IA 재편 — 8개 → 3개

### 기존
```
대시보드 / 주간현황 / 이익 / 수입 / 지출 / 메뉴분석 / 업로드 / 설정
```
점장 피드백: "뭔가뭔가 비슷한 것들인데 쪼개져 있는 느낌. 아 이거 어디였지? 하면서 이것저것 들어가봐야 한다"

### 4개 중간안 (버림)
```
현황 (대시보드+주간)
손익 (수입+지출+이익)
메뉴분석
관리 (업로드+설정)
```
문제: 종합(이익)이 시간축으로는 주간·월간과 같은 성격인데 손익에 묶여있어 혼란. 메뉴분석이 매출 관점과 겹침.

### 최종안 — 시간축 vs 원인분석으로 재정렬
```
현황 = 월간 / 주간 / 종합    (시간축만 다른 같은 숫자)
분석 = 매출 / 지출 / 메뉴    (원인 파기)
관리 = 업로드 / 설정
```

사고 흐름:
- "얼마야?" → **현황** (원하는 시간 단위로 점프)
- "왜 그런가?" → **분석** (매출·지출·메뉴 관점별로 뜯어보기)
- "데이터 건드리기" → **관리**

### URL 유지, PageTabs 컴포넌트로 서브 내비
- 기존 경로(`/`, `/weekly`, `/profit`, `/income`, `/expenses`, `/menu-analysis`, `/upload`, `/settings`) 그대로 — 북마크 안 깨짐
- `components/ui/PageTabs.tsx` 신설. `group` prop으로 그룹 지정하면 그룹 내 탭 자동 렌더
- 페이지 간 이동 시 `?year=&month=` 같은 쿼리 유지 (기간 필터 이어감)

### 사이드바 구현
- `NAV_ITEMS` 3개, 각 `group: string[]`로 포함 경로 명시
- `isActive(group)`가 현재 pathname이 그룹 안에 있는지 판정 → 그룹 전체 활성화
- 탭 이름 "대시보드" 대신 **"월간"** 으로 통일 (시간축 일관성)

| 이슈 | 원인 | 해결 |

| 재료비(현금) 0원으로 뜸 | `UPDATE ... SET category = 'ingredients_card'`로 기존 ingredients 전부 카드로 밀어버림 | 통장 재업로드로 새 파서가 수취인 기준 재분류 (`(tx_time, balance_after)` upsert로 덮어쓰기) |

## 7. 메뉴분석·죽은메뉴·디저트 단위·핫아이스 — 아직 구현 안 함

점장 미팅에서 합의됐고 파싱·IA·설정 쪽 먼저 땜빵했음. 다음 세션 작업:
- [ ] POS 상품명 핫/아이스 suffix 정규화 (쿼리 단계)
- [ ] 죽은 메뉴 기준 — 전체 기간 10잔 미만 + 마지막 판매일 뱃지

### 원가 계산 (점장과 추가 논의 예정)
두 방향 중 하나로 정해지면 시작:
- (i) 거래내역에 수량을 쓰기 — 지속성 낮음
- (ii) 고정 수량 매칭 — 거래처별로 "한 번 살 때 10kg"처럼 룰 등록 → 점장이 한 말("원두 1kg 한 봉지, 한잔 18-19g, 한 번 살 때 10봉")과 정확히 일치

## 8. 느낀 점

- **"왜 그 카테고리인지"의 근거가 항상 수취인**: 파서가 `memo(적요)`만 보고 `counterpart(수취인)`을 DB에 저장 안 해온 게 이번에 재분류 못 하는 원인이 됨. 향후 `monthly_expenses`에 `counterpart` 컬럼 추가 고려 (재업로드 없이도 룰 변경 적용 가능해짐).
- **예금주 본인 거래는 대부분 무관**: 박기선 건 실데이터 파보니 수취인이 "박기선" 정확히 일치하는 건 4건 중 2건뿐. 나머지 2건은 지방세였음. "이름만 포함되면 제외"는 과한 규칙이었음.
- **토글 vs 수동 조정의 경계 설계가 깔끔했음**:
  - 반복적·패턴화된 모호 그룹 → 토글
  - 일회성 예외(장천섭 100만원치 음료 제공 같은) → 수동 조정
  - 두 메커니즘이 상호보완이라 미래 확장에 유리.
- **IA 재편의 감(感)**: "대시보드·주간현황·이익"이 쪼개져 있던 게 점장을 혼란스럽게 하던 본질. 셋 다 "얼마야?"의 답이고 시간축만 다름. 묶고 나니 "왜?"를 담당하는 분석 탭의 역할이 뚜렷해짐. IA는 **사용자가 그 페이지에서 던지는 질문**으로 그룹핑하는 게 답.
- **URL 유지 + PageTabs 조합**: Next.js 중첩 레이아웃까진 안 써도 충분. 기존 경로 안 깨면서도 그룹 네비만 얹음. 오버엔지니어링 회피.
